# Styling

## Overview

Styling is exclusively Tailwind CSS v4 utility classes used in markup, with a "terminal skin" visual identity layered on via theme tokens and Starlight CSS variable overrides. There is one stylesheet — `src/styles/global.css` — and it is a tokens-only file: `@fontsource/jetbrains-mono` weight imports (400/500/600/700), `@import "tailwindcss";`, a Tailwind v4 `@theme inline` block declaring color tokens, the `--font-mono` token, the `--animate-blink` token plus `@keyframes blink`, and two `:root` blocks (`:root` for light, `:root[data-theme="dark"]` for dark) that bind both `--c-*` palette variables and Starlight's `--sl-*` integration variables. No component-scoped CSS, no CSS-in-JS, no PostCSS configuration beyond the Tailwind Vite plugin.

The palette is "phosphor-amber + sage + teal" — warm near-black dark theme and warm-cream "paper terminal" light theme. Dark is the primary mode. Starlight's built-in toggle sets `data-theme="dark"` on `<html>`; both themes are required to render correctly (Constitution V).

All terminal flourishes on the home page (blinking cursor, `$` prompts, `##` sage prefixes, ASCII rules, code-block corner labels, tree-list rows) are expressed as Tailwind utilities in markup — either inline in MDX or composed inside small prop-only Astro components in `src/components/`. None of them are CSS class rules in `global.css`.

## Requirements

- All styling MUST use Tailwind utility classes directly in markup. Custom CSS class selectors (`.classname { ... }`) SHALL NOT be defined in `src/styles/global.css` or any other stylesheet.
- `src/styles/global.css` MUST contain only: (a) `@import` directives for `@fontsource/jetbrains-mono` weights and Tailwind, (b) a `@theme` block declaring color/font/animation tokens, and (c) `:root` and `:root[data-theme="dark"]` blocks binding `--c-*` palette variables plus Starlight `--sl-*` integration variables. It SHALL NOT contain component-scoped rules.
- Components under `src/components/` (Astro files) MAY use Tailwind utility classes in their markup, but MUST NOT contain `<style>` blocks with class-based CSS rules. Constitution III applies to components as well as the stylesheet — the constitutional distinction is "Astro component vs CSS class", not "stylesheet vs file". A small prop-only component that composes Tailwind utilities is allowed (and encouraged for dedup); a `<style>` block defining `.terminal-cursor { ... }` is not.
- Tailwind v4 MUST be integrated via the `@tailwindcss/vite` plugin in `astro.config.mjs`, registered through `vite.plugins`.
- The stylesheet MUST be registered with Starlight via `customCss: ['./src/styles/global.css']` so it is included in the build output.
- Every visible UI element MUST render correctly in both light and dark modes. Token swap is driven by the `:root` / `:root[data-theme="dark"]` blocks — `--c-*` values change between themes; utilities like `bg-bg`, `text-fg`, `text-accent` resolve to the active theme automatically via the `@theme inline` proxy layer. `dark:` Tailwind variants are unnecessary for color tokens and SHOULD NOT be used for them.
- The H1 blinking cursor MUST respect `prefers-reduced-motion` — the `animate-blink` utility is paired with `motion-reduce:animate-none` so reduced-motion users see a static block, not an animation.
- Starlight chrome (sidebar, TOC, search modal, doc-template tool pages) MUST inherit the new look via Starlight CSS variable overrides — never by forking Starlight components. The 12-variable surface in `global.css` (`--sl-font`, `--sl-font-mono`, `--sl-color-bg`, `--sl-color-bg-sidebar`, `--sl-color-bg-nav`, `--sl-color-text`, `--sl-color-text-accent`, `--sl-color-accent`, `--sl-color-accent-low`, `--sl-color-accent-high`, `--sl-color-hairline`, `--sl-color-hairline-light`) is the minimum surface; additional variables MAY be added if visual review reveals default-theme leakage.
- Diagrams MUST be pre-rendered to SVG at author-time and committed under `public/diagrams/` (one file per theme: `*-light.svg`, `*-dark.svg`). The runtime mermaid library SHALL NOT be a dependency. See [diagrams](./diagrams.md).

## Design Decisions

- **Tailwind v4 (not v3).** v4 is the current major; uses the new Vite-plugin architecture, no separate `tailwind.config.js` required for the default config. Tokens declared in `@theme` become utility classes automatically (e.g., `--color-accent` → `bg-accent`, `text-accent`, `border-accent`). Rationale: green-field project, no migration burden; theme-as-tokens is the v4-native ergonomic.
- **`@theme inline` with a `--c-*` proxy layer, not direct hex values in `@theme`.** Tailwind v4's `@theme` block normally inlines values into the generated utility CSS at build time. With `@theme inline`, Tailwind instead resolves utilities to `var(--color-*)` references that hit the cascade at runtime — so the `:root` / `:root[data-theme="dark"]` swap actually drives a theme change without rebuilding. The `--c-*` proxy (`--color-bg: var(--c-bg)`, etc.) is what makes the swap work: `--color-*` is the Tailwind contract; `--c-*` is the value that flips per theme. Rationale: Tailwind v4 documents this exact pattern for runtime-swappable themes; doing it without `inline` would freeze the colors at build time and break the toggle.
- **`@fontsource/jetbrains-mono` (self-hosted) over Google Fonts `<link>`.** Self-hosted woff2s eliminate a runtime third-party request to `fonts.googleapis.com`, give deterministic offline builds, ship inside the same GitHub Pages cache as the rest of the site, and avoid sending visitor IPs to Google. The dependency cost is one npm package (~120KB across 4 weights). Rejected: Google Fonts `<link>` — adds a third-party DNS lookup before first paint, runtime CDN dependency, and a privacy footprint that's hard to audit. Constitution IV (minimal dependencies) is satisfied because typography IS the redesign — without a font swap, no other change matters.
- **Token-based theming over `dark:` Tailwind variants for color.** Once `bg-bg` resolves through `--color-bg → --c-bg`, the same utility yields different pixels in dark vs light without authoring `dark:` modifiers. This keeps the markup smaller and more readable, and centralizes palette changes in one place (`global.css`). `dark:` is still appropriate for non-color toggles (e.g., `dark:invert` on a logo); for colors, prefer tokens.
- **Components in `src/components/` may compose Tailwind utilities, but MUST NOT contain class-based CSS rules.** When a utility chain repeats 4+ times across the codebase, extract a small prop-only Astro component (e.g., `HRule`, `ToolRow`) — this is Constitution-III-compliant because the component's source is `<div class="text-fg-faint ...">` (utilities in markup), not `.hrule { color: var(...); ... }` (a CSS class rule). Rejected: defining `.terminal-cursor`, `.shell-prompt`, `.tree-row`, etc. as CSS classes in `global.css`. Violates Constitution III, fragments styling logic between two files, and creates a hidden vocabulary contributors must learn before editing the page.
- **Starlight CSS variable overrides for chrome (sidebar/TOC/search/tool pages).** Setting `--sl-font`, `--sl-color-bg`, `--sl-color-text`, `--sl-color-accent`, etc. propagates to every Starlight-rendered surface without touching component templates. Tool subpages get the new palette + JetBrains Mono "for free" with zero per-page edits. Rejected: forking Starlight components to inject our own classes — high maintenance burden, breaks on Starlight upgrades, unnecessary for visual-only changes.
- **No custom CSS architecture (BEM, CSS Modules, CSS-in-JS).** This is a single-page marketing site with a small `/tools/` directory. Introducing a CSS architecture would be unjustified complexity for the surface area; Tailwind utilities + 10 color tokens + 4 small components is the right size.

## Integration Points

- `astro.config.mjs`: `vite.plugins: [tailwindcss()]` — wires Tailwind into the Vite build
- `astro.config.mjs`: `starlight({ customCss: ['./src/styles/global.css'] })` — includes the stylesheet in every page
- `astro.config.mjs`: `starlight({ components: { Hero: './src/components/Hero.astro' } })` — registers the Hero component override so the home page can render the shellpath + canonical `<h1>` + blinking cursor + prompt-prefixed CTAs in a single Starlight-rendered hero. See [astro-config](./astro-config.md).
- `src/styles/global.css`: the entry file — tokens-only by constitutional mandate; deviations require justification against [the constitution](../../../fab/project/constitution.md) Principle III
- `src/components/Hero.astro`, `HRule.astro`, `ToolRow.astro`: prop-only Astro components composed of Tailwind utilities — they dedup utility chains across multiple call sites without introducing CSS classes. See [site-architecture](./site-architecture.md) for the full component layout.
